# HomeClaw 设计文档

## 一、产品定位

用 AI + 工作流取代 HomeAssistant 的配置体系，让普通家庭用户通过对话管理智能家居，无需任何技术背景。

**核心问题诊断：HomeAssistant 为什么对小白不友好？**
- 设备接入复杂 — 需要手动配置 YAML、集成插件、调试协议（MQTT/Zigbee/Z-Wave）
- 自动化配置复杂 — 条件触发规则是代码思维，非自然语言
- 概念爆炸 — Entity/Domain/Service/Automation/Script/Blueprint... 术语门槛高
- 反馈不透明 — 出错了不知道为什么，日志太技术化

---

## 二、核心决策

| 维度 | 决策 |
|---|---|
| 与 HA 关系 | 完全替代，不依赖 HA，直接对接设备原生协议 |
| 设备生态 | 米家 / 涂鸦 / HomeKit / Matter，Skill 插件化 |
| 部署硬件 | 树莓派 / NAS / 旁路由 / 超轻量板卡 全兼容 |
| 网络策略 | 本地优先，云端做补充，断网仍可执行已有工作流 |
| AI 模型 | 云端大模型 + 本地模型（Ollama），用户自选 |
| 交互渠道 | Web UI（启动向导）+ Telegram / 微信 + 小爱同学 / 天猫精灵 |
| 用户体系 | 多用户，家庭成员权限隔离 |
| 与 PicoClaw 关系 | 独立 fork，思路对齐后决定复用模块方式 |

---

## 三、架构分层

```
[交互层]
  Web UI（启动向导 + 基础配置）
  Telegram / 微信（日常控制）
  小爱同学 / 天猫精灵（语音，识别在厂商云，控制走本地）

[AI 决策层]
  小模型（本地，实时）→ 意图分类 → 选择工作流
  大模型（云端/本地大模型，按需）→ 生成工作流 / 处理复杂意图

[工作流引擎层]
  工作流定义：轻量 JSON/YAML
  触发源：用户对话 / 设备事件 / 定时任务
  执行：HomeClaw 节点（树莓派 / NAS / 旁路由）

[设备接入层]
  Skill 插件化，统一接口契约
  米家 Skill / 涂鸦 Skill / HomeKit Skill / Matter Skill / ...
  本地协议直连，不经过厂商云（涂鸦部分设备除外）

[硬件节点层]
  HomeClaw 节点：树莓派 / NAS / 旁路由 / 超轻量板卡
```

---

## 四、AI 分工

| 角色 | 触发时机 | 负责什么 |
|---|---|---|
| 大模型 | 首次配置、生成自动化、复杂问题 | 理解意图、生成工作流定义 |
| 小模型 | 每次用户交互、设备事件 | 意图分类、选择匹配的工作流 |
| 工作流引擎 | 小模型路由后 | 本地直接执行，不再调 AI |

**AI 价值优先级：**

```
P0  日常控制 → "把客厅灯关掉"
P1  设备接入引导 → "我买了个米家温湿度计，帮我接进来"
P2  自动化规则 → "每天晚上 10 点关所有灯"
P3  习惯学习 → "你最近总是 23 点才关灯，要不要设成自动的？"
```

---

## 五、工作流机制

- **格式**：轻量 JSON/YAML，由大模型生成，存储在本地
- **触发源**：
  - 用户对话（主动控制）
  - 设备事件（传感器状态变化 → 自动执行）
  - 定时任务（cron 触发）
- **执行位置**：主节点
- **断网行为**：已有工作流在主节点离线可执行

---

## 六、Skill 标准契约

每个生态 Skill 必须实现以下统一接口：

| 接口 | 说明 |
|---|---|
| 发现 | 扫描局域网内该生态的设备 |
| 注册 | 把发现的设备登记到 HomeClaw 设备注册表 |
| 读状态 | 查询设备当前状态（温度 / 亮度 / 开关...） |
| 写控制 | 执行控制指令（开灯 / 调温 / 锁门...） |
| 订阅变化 | 设备状态变更时主动推送（事件驱动） |
| 元数据 | 描述设备有哪些能力（供 AI 理解和调用） |

**各生态本地控制可行性：**

| 生态 | 本地控制 | 说明 |
|---|---|---|
| 米家 | 可行 | miot-spec 支持局域网直连（miio 协议） |
| 涂鸦 | 部分可行 | 本地 Local API + WebSocket，部分设备强依赖云 |
| HomeKit | 完全本地 | HAP 协议天然本地 |
| Matter | 完全本地 | 标准设计本地优先，支持订阅 |

---

## 七、用户体系

- **管理员**：负责设备接入、添加家庭成员、管理权限
- **普通成员**：按权限控制设备和空间
- **访客**（待定）：临时权限

---

## 八、启动流程

```
首次启动 Web UI
  → 设置用户名密码
  → 配置 AI 模型（大模型 API Key 或本地 Ollama 地址）
  → 扫描接入设备（Chat 方式 或 页面方式二选一）
  → 绑定控制渠道（Telegram / 微信等）
  → 进入日常使用
```

---

## 九、语音平台策略

小爱同学 / 天猫精灵的接入存在本质约束：
- 语音识别在厂商云端完成（无法绕开）
- 控制指令下发后走本地执行（保持本地优先）
- 语音识别环节接受依赖厂商云，设备控制仍走本地

# 十、用户故事

### US-01 首次启动与初始化

**作为**新用户，**我希望**首次访问 Web UI 时被引导完成所有初始配置，**以便**不需要阅读任何文档就能上手。

```
场景：
1. 打开浏览器访问 HomeClaw Web UI
2. 设置管理员用户名和密码
3. 填写 AI 模型配置（API Key 或本地 Ollama 地址）
4. 系统验证模型可用后，进入 Chat 引导流程
```

**作为**管理员，**我希望**通过对话方式配置 Telegram/QQ 等渠道，**以便**后续可以通过这些渠道控制家里的设备。

```
场景：
1. HomeClaw 通过 Chat 引导配置 Telegram Bot Token
2. 配置完成后，用户可通过 Telegram 与 HomeClaw 对话
3. Web UI Chat 始终可用，不依赖第三方渠道
4. 所有渠道共享同一份家庭数据和对话上下文
```

**作为**管理员，**我希望**用自然语言描述家里的空间结构，**以便** HomeClaw 知道设备归属在哪个位置。

```
场景：
HomeClaw："你家有哪些房间？"
用户："一楼有客厅、餐厅、厨房，二楼有主卧、儿童房、书房"
HomeClaw 自动结构化存储，支持任意层级粒度
```

**作为**管理员，**我希望** HomeClaw 帮我扫描家里的设备并自动归属到对应房间，**以便**不需要手动一个个配置。

```
场景：
1.  HomeClaw："你家用的是什么品牌？"
2.  用户："小米"
3.  加载米家 Skill，引导登录获取 token
4.  扫描局域网内所有米家设备
5.  设备命名含房间关键词 → AI 直接推断归属
6.  无法推断 → 询问用户："这个设备叫 XX，它在哪个房间？"
7.  用户也不确定 → 开关确认法（开/关设备让用户观察）
8.  传感器类设备 → 跳过开关法，直接询问或读数值辅助判断
9.  一个品牌扫描完再问下一个品牌
10. 目标：所有设备都有明确的空间归属
```

---

### US-05 添加家庭成员

**作为**管理员，**我希望**为家庭成员创建账号并设定权限，**以便**每个人只能控制自己有权限的设备。

```
场景：
1. HomeClaw："你家还有其他成员吗？"
2. 用户："老婆、两个小孩"
3. 为每个成员设定可控制的房间范围
   （如：小孩只能控制自己卧室，不能控制大门/摄像头）
4. 可选：为成员绑定 Telegram/QQ 账号
   → 管理员生成验证码 → 成员发给 Bot 完成身份绑定
5. 不绑定渠道的成员只能通过 Web UI 使用
```

---

### US-06 动态添加新设备

**作为**用户，**我希望**通过语音或对话随时添加新买的设备，**以便**不需要打开配置界面。

```
场景：
用户对小米音箱说："小爱，告诉 HomeClaw 我买了个新的小米摄像头"
  → HomeClaw 收到请求，触发米家增量扫描
  → 发现新摄像头，执行设备归属流程
  → 同时通过语音播报 + Telegram + Web UI 通知反馈结果
  → 用户在任意渠道回复确认后，其他渠道标记为已处理
```

---

### US-07 空间感知的设备控制

**作为**用户，**我希望**说「开灯」时只影响我当前所在房间的灯，**以便**不需要每次都指定位置。

```
场景 A（音箱控制，有空间信息）：
  用小爱同学说"开灯" → 打开音箱所在房间的灯

场景 B（Telegram 控制，有身份信息）：
  发消息"开灯" → 推断该成员所在/常驻房间 → 打开对应房间的灯
  语义模糊时追问："你是要开哪个房间的灯？"

场景 C（明确位置）：
  "打开客厅的灯" → 直接操作客厅的灯

场景 D（全局操作）：
  "关掉所有灯" → 遍历关闭所有房间的灯
```

---

### US-08 场景工作流触发

**作为**用户，**我希望**用一句话触发一组设备操作，**以便**不需要逐个控制设备。

```
场景：
用户说："我要睡觉了"
  → 小模型匹配「睡眠场景」工作流（预生成 或 实时由大模型生成后存储）
  → 执行：关闭当前房间的灯
  → 小朋友说："保留台灯"
  → HomeClaw 识别为对上一工作流的修正
  → 撤销台灯关闭，追加「延时 1 小时关闭台灯」
```

**工作流生成双模式：**
- 预生成：用户主动说「帮我设置睡眠场景」→ 大模型生成存储
- 实时生成：小模型无法匹配 → 上报大模型 → 生成执行 → 自动存储

---

### US-09 习惯学习与个性化

**作为**用户，**我希望** HomeClaw 记住我的偏好，**以便**常用的个性化操作不需要每次重复说明。

```
场景：
小朋友在睡眠场景后要求保留台灯
  → HomeClaw 询问："要把这个偏好保存下来吗？"
  → 确认 → 为该成员创建个性化工作流变体
  → 下次触发睡眠场景时，自动保留台灯
  → 拒绝 → 本次一次性执行，不记录
```

**用户识别规则：**
- 小爱/天猫精灵：以音箱所在房间为空间参考，声纹识别（部分设备支持）
- Telegram/QQ：通过绑定账号识别成员身份
- Web UI：当前登录用户
- 无法确认身份或位置时：追问用户

---

### US-10 修改空间 / 设备 / 成员

**作为**管理员，**我希望**随时通过对话或页面修改已有配置，**以便**家庭情况变化时能及时更新。

```
场景示例：
- "把书房改名为工作室"
- "把走廊的灯移到客厅"
- "给老婆开通卧室摄像头的权限"
- "我买了个新的涂鸦插座，帮我接进来"
```
